================================================================================
ARCHITECTURE OVERVIEW
================================================================================

The pipeline is a state machine with five sequential agent stages, a routing
decision, an optional pause for human review, and a finalize step:

    classify -> extract -> compare -> score -> verify -> {hitl?} -> finalize

Each agent reads a typed dict-shaped state and returns an updated state. They
never call each other directly — they share data through the LangGraph state
graph. That separation is what makes the system testable, swappable, and
auditable.

The five agents:

  classify   — identifies contract type (NDA, MSA, SaaS, etc.)
  extract    — pulls the high-leverage clauses verbatim from the contract
  compare    — for each clause, retrieves the matching playbook position via
               RAG and asks the LLM to compare them
  score      — assigns a 0-100 risk score per clause and an overall score
  verify     — independently checks the agents' work for hallucination
               (fabricated quotes, unsupported rationales)

The HITL gate is implemented as a routing edge after the verifier. If the
overall risk score is at or above the threshold (default 60) OR the verifier
raised a high-severity finding, the graph pauses before the `hitl` node. The
UI collects a human decision (approve / reject / modify, with optional clause
score overrides), writes it back into graph state, and the graph resumes.

The infrastructure under all of this:

  state.py     — typed dicts for the graph state and major sub-objects
  audit.py     — SQLite-backed structured event log
  progress.py  — singleton reporter agents emit into for live UI updates
  llm.py       — single chokepoint for all LLM calls (audit + retries + emits)
  rag.py       — ChromaDB-backed playbook retrieval


================================================================================
KEY CONCEPTS — RECURRING PATTERNS YOU'LL SEE THROUGHOUT
================================================================================

These patterns appear in multiple files. Rather than re-explain them every time,
they are introduced here once.

(1) THE STATE PATTERN

Every agent has the signature:

    def agent(state: ReviewState) -> ReviewState:
        # read whatever it needs from `state`
        # do its work
        return {**state, "new_key": new_value}

The `{**state, ...}` pattern means "return a new dict containing everything in
state PLUS my new keys." This is functional-style state evolution. The agent
never mutates `state` in place; it produces a new state dict that LangGraph then
threads to the next node.

Why this matters for the demo: an interviewer asking "how do agents
communicate?" can hear "they don't — they read and write a shared typed state
object that the graph orchestrates between them." That's a cleaner architecture
than agent-to-agent message passing.

(2) AUDIT-FIRST DESIGN

Every LLM call goes through src/llm.py:call_llm, which logs the call to the
audit DB before returning. The agents do not call the Anthropic SDK directly.
This is the difference between "we have an audit log" and "we have an audit
log that you can trust." If a developer ever called the SDK directly,
ungoverned, the audit would have a hole. The chokepoint design makes that
impossible.

If asked "how do you guarantee the audit log is complete?" the answer is:
"there is exactly one function in the codebase that calls the LLM, and that
function logs every call. Anything else is a code review failure, not an
architectural one."

(3) PROMPT VERSIONING

Every prompt has a version tag (currently 'v1') stored in config.py. Every
LLM call records the prompt version that produced it in the audit DB. This is
what makes "what prompt generation produced these outputs?" answerable.

When you change a prompt — even a single word — bump its version in
PROMPT_VERSIONS. The audit log will then let you filter runs by prompt
generation. This is the operational discipline that distinguishes a real
governance program from a checkbox.

(4) PER-CLAUSE LOOPS

Three agents (compare, score, verify) loop over the extracted clauses, making
one LLM call per clause. This is deliberate: it gives finer-grained outputs,
better RAG retrieval (one query per clause type), and easier per-clause
verification. The cost is N times more LLM calls per contract.

For the aggressive sample contract with ~21 extracted clauses, that translates
to ~66 LLM calls per run and roughly $0.33 in token cost. That's the live cost
your UI shows during a run, and it's the number to memorize for interview
discussion.

(5) HITL VIA INTERRUPT_BEFORE

LangGraph's `interrupt_before=["hitl"]` tells the graph to pause its execution
before the named node runs and persist its state to a checkpointer. The UI can
then read the paused state, present it to a human, collect a decision, write
the decision back into state via `graph.update_state()`, and resume execution
by calling `graph.stream(None, ...)` with the same thread_id.

This is the architectural pattern that lets a paused review survive a browser
refresh, a process restart, or a transfer between reviewers. The graph state
lives in SQLite, not in memory.

(6) THE PROGRESS REPORTER

The progress reporter (src/progress.py) is a singleton that agents and the LLM
wrapper emit events into. The UI registers a callback that translates events
into Streamlit updates. If no callback is registered (CLI runs, tests,
notebooks), emits are silent no-ops.

This decoupling is the right separation: progress is observability, not
control flow. Agents work the same in every environment; only the UI knows
how to render their progress.

ANNOTATION: The version specifiers use `>=` rather than pinning exact versions.
This is a portfolio-project tradeoff: the project gets newer features and
security patches automatically, at the cost of occasional breakage when an
upstream library makes a backwards-incompatible change. For a production
deployment you would pin versions exactly (using `pip freeze > requirements.txt`
after a known-good install) and update them deliberately.

Each dependency is here for a specific reason:

  anthropic                     — Claude API client; what call_llm() wraps.
  langgraph                     — The state machine framework. Provides the
                                  StateGraph, conditional edges, and streaming
                                  primitives we use in orchestrator.py.
  langgraph-checkpoint-sqlite   — SQLite checkpointer for LangGraph. Required
                                  to persist paused graph state across
                                  processes. Without this, HITL pause/resume
                                  would only work within a single process.
  chromadb                      — Embedded vector database. Stores the
                                  embedded playbook for RAG retrieval. No
                                  server needed; runs in-process.
  sentence-transformers         — Local embedding model (all-MiniLM-L6-v2).
                                  ~80 MB, runs on CPU, downloaded once on first
                                  use. Used to embed playbook sections and
                                  clause queries.
  streamlit                     — UI framework. Renders the upload, review,
                                  and done pages.
  pydantic                      — Validation library used by some of the
                                  upstream packages. Not directly imported by
                                  our code, but listed because pinning the
                                  major version prevents a transitive break.
  python-dotenv                 — Reads .env file into environment variables
                                  at startup. Called from config.py.
  tenacity                      — Retry/backoff library wrapped around the LLM
                                  call to survive transient API failures.

--------------------------------------------------------------------------------
FILE: src/state.py
--------------------------------------------------------------------------------

PURPOSE: Defines the typed dictionaries that flow through the LangGraph
workflow. These are not validated at runtime (they are TypedDicts, not
Pydantic models) — they exist for static type-checking and as documentation
of what each stage produces and consumes.

----------------------------------------------------------------
[Block 1: imports and Clause]
----------------------------------------------------------------

    from __future__ import annotations
    from typing import TypedDict, Optional, Any


    class Clause(TypedDict, total=False):
        clause_id: str
        clause_type: str
        text: str
        location_hint: str
        playbook_matches: list[dict[str, Any]]
        comparison: dict[str, Any]
        risk_score: int
        risk_rationale: str

ANNOTATION: `total=False` means every field is optional. That is deliberate —
the Clause object accumulates fields as it moves through the pipeline:

  - The extractor populates clause_id, clause_type, text, location_hint
  - The compare agent adds playbook_matches and comparison
  - The risk scorer adds risk_score and risk_rationale

A Clause being passed to the compare agent has only the first four fields;
one returned from the score agent has all of them. Total=False captures that
honestly without forcing every stage to write every field.

----------------------------------------------------------------
[Block 2: VerifierFinding]
----------------------------------------------------------------

    class VerifierFinding(TypedDict):
        clause_id: str
        issue: str
        severity: str  # low | medium | high
        quote_from_ai: str
        quote_from_source: Optional[str]

ANNOTATION: This one does NOT have total=False — every field is required when
a finding is created. The verifier either produces a complete finding or
none at all.

`quote_from_source` is Optional because the mechanical fidelity check
(detecting fabricated quotes) sets it to None: there is no faithful source
quote to extract because the AI's quote wasn't in the source.

----------------------------------------------------------------
[Block 3: HITLDecision]
----------------------------------------------------------------

    class HITLDecision(TypedDict, total=False):
        decision: str  # approve | reject | modify
        reviewer: str
        notes: str
        modified_clauses: dict[str, dict[str, Any]]

ANNOTATION: Captures everything the human reviewer submitted. `modified_clauses`
is a map of clause_id to override fields — most commonly `{"risk_score": 96}`
to bump a per-clause score the AI undershot.

This dict is what the orchestrator reads and applies in _hitl_node. It is
also what gets logged to the audit DB as a 'human_decision' event, making
human overrides as auditable as AI outputs.

----------------------------------------------------------------
[Block 4: ReviewState]
----------------------------------------------------------------

    class ReviewState(TypedDict, total=False):
        run_id: str
        contract_name: str
        contract_text: str
        contract_type: str
        contract_type_confidence: float
        clauses: list[Clause]
        overall_risk_score: int
        overall_risk_rationale: str
        verifier_findings: list[VerifierFinding]
        requires_human_review: bool
        hitl_decision: Optional[HITLDecision]
        final_report: Optional[str]
        error: Optional[str]

ANNOTATION: The top-level state object that LangGraph threads between nodes.
Every agent reads from this and writes back to it. Reading the field list in
order is roughly the timeline of a run:

  - run_id, contract_name, contract_text  →  set by app.py before the graph runs
  - contract_type, contract_type_confidence  →  set by classifier
  - clauses (list of Clause)  →  populated by extractor, enriched by compare
                                  and score
  - overall_risk_score, overall_risk_rationale  →  set by score
  - verifier_findings, requires_human_review  →  set by verify
  - hitl_decision  →  set by app.py via graph.update_state() if HITL triggered
  - final_report  →  set by the finalize node
  - error  →  reserved; not currently used


--------------------------------------------------------------------------------
FILE: src/audit.py
--------------------------------------------------------------------------------

PURPOSE: SQLite-backed structured audit log. Every LLM call, retrieval, and
human decision is written here with enough metadata to reconstruct the run.
This is the operational governance layer — what the JD calls "audit trails
that protect clients from liability."

----------------------------------------------------------------
[Block 1: schema]
----------------------------------------------------------------

    SCHEMA = """
    CREATE TABLE IF NOT EXISTS runs (
        run_id TEXT PRIMARY KEY,
        contract_name TEXT,
        started_at TEXT NOT NULL,
        completed_at TEXT,
        final_status TEXT,
        final_risk_score INTEGER
    );

    CREATE TABLE IF NOT EXISTS events (
        event_id TEXT PRIMARY KEY,
        run_id TEXT NOT NULL,
        timestamp TEXT NOT NULL,
        stage TEXT NOT NULL,
        event_type TEXT NOT NULL,  -- llm_call | retrieval | human_decision | error
        model TEXT,
        prompt_version TEXT,
        input_tokens INTEGER,
        output_tokens INTEGER,
        payload_json TEXT NOT NULL,
        FOREIGN KEY (run_id) REFERENCES runs(run_id)
    );

    CREATE INDEX IF NOT EXISTS idx_events_run ON events(run_id);
    CREATE INDEX IF NOT EXISTS idx_events_stage ON events(stage);
    """

ANNOTATION: Two tables, kept intentionally simple.

  runs    — one row per contract review. Tracks start time, end time, final
            decision (approve/reject/modify), and final risk score. The fields
            are nullable for completed_at, final_status, and final_risk_score
            because a row is INSERTed at run start and UPDATEd at run end.

  events  — many rows per run, each representing a discrete observable event.
            Three event types exist today: llm_call, retrieval, and
            human_decision. The schema reserves room for 'error' but no agent
            currently logs error events directly (errors propagate as raised
            exceptions and Streamlit catches them).

Indexes on run_id and stage make the most common queries fast: "show me all
events for run X" and "show me all classifier calls across all runs."

`payload_json` is a TEXT column holding the full structured payload as JSON.
This is deliberate — not every event has the same shape, and you don't want
to invent a column-per-event-type schema. Querying inside the JSON is a bit
clunky in raw SQLite but works fine; for serious analytics you would ETL the
audit DB into a proper warehouse periodically.

If asked "why SQLite and not Postgres?" the answer is: SQLite is the simplest
thing that supports the durability and querying we need for a single-tenant
demo. The schema is deliberately portable — moving it to Postgres is a
connection-string change, not a rewrite.

----------------------------------------------------------------
[Block 2: connection helper and init]
----------------------------------------------------------------

    @contextmanager
    def _conn():
        conn = sqlite3.connect(str(AUDIT_DB_PATH))
        conn.row_factory = sqlite3.Row
        try:
            yield conn
            conn.commit()
        finally:
            conn.close()


    def init_db() -> None:
        with _conn() as c:
            c.executescript(SCHEMA)

ANNOTATION: `_conn` is a context manager that handles open / commit / close
in one place. `sqlite3.Row` makes results dict-like (you can do `row['stage']`
instead of `row[3]`).

`init_db()` runs the schema. It uses `CREATE TABLE IF NOT EXISTS` so calling
it on an existing DB is harmless. start_run() calls init_db() automatically,
so you never need to call it explicitly.

----------------------------------------------------------------
[Block 3: run lifecycle]
----------------------------------------------------------------

    def start_run(contract_name: str) -> str:
        init_db()
        run_id = str(uuid.uuid4())
        with _conn() as c:
            c.execute(
                "INSERT INTO runs (run_id, contract_name, started_at) VALUES (?, ?, ?)",
                (run_id, contract_name, _now()),
            )
        return run_id


    def complete_run(run_id: str, final_status: str, final_risk_score: Optional[int]) -> None:
        with _conn() as c:
            c.execute(
                "UPDATE runs SET completed_at = ?, final_status = ?, final_risk_score = ? WHERE run_id = ?",
                (_now(), final_status, final_risk_score, run_id),
            )

ANNOTATION: A run's lifecycle has two writes:

  start_run    — INSERT a row with the run_id (UUID) and start timestamp.
                  Returns the run_id, which becomes the foreign key for every
                  event the run produces.

  complete_run — UPDATE the row when the human submits a final decision.
                  Sets the completion timestamp, the final status (the
                  HITL decision verb: approve/reject/modify), and the final
                  risk score.

A run that crashes halfway through will have a row with NULL completed_at —
a useful signal that the run never finished cleanly.

----------------------------------------------------------------
[Block 4: log_event]
----------------------------------------------------------------

    def log_event(
        run_id: str,
        stage: str,
        event_type: str,
        payload: dict[str, Any],
        model: Optional[str] = None,
        prompt_version: Optional[str] = None,
        input_tokens: Optional[int] = None,
        output_tokens: Optional[int] = None,
    ) -> str:
        event_id = str(uuid.uuid4())
        with _conn() as c:
            c.execute(
                """INSERT INTO events
                   (event_id, run_id, timestamp, stage, event_type, model,
                    prompt_version, input_tokens, output_tokens, payload_json)
                   VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?)""",
                (
                    event_id,
                    run_id,
                    _now(),
                    stage,
                    event_type,
                    model,
                    prompt_version,
                    input_tokens,
                    output_tokens,
                    json.dumps(payload, default=str),
                ),
            )
        return event_id

ANNOTATION: This is the single function that writes events to the audit DB.
Called from src/llm.py on every LLM call, from src/rag.py on every retrieval,
and from the orchestrator's hitl node when a human decision arrives.

`json.dumps(payload, default=str)` handles non-JSON-serializable objects
(datetime, UUID, etc.) by stringifying them. That is forgiving for a logging
function — you would rather log a slightly degraded payload than crash the
pipeline.

The event_id is a UUID that lets downstream tooling reference a specific
event uniquely (e.g. "the verifier finding from event abc-123 was the false
positive that caused this contract to escalate"). Useful for incident
forensics.

----------------------------------------------------------------
[Block 5: query helpers]
----------------------------------------------------------------

    def get_events_for_run(run_id: str) -> list[dict[str, Any]]:
        with _conn() as c:
            rows = c.execute(
                "SELECT * FROM events WHERE run_id = ? ORDER BY timestamp ASC",
                (run_id,),
            ).fetchall()
        return [dict(r) for r in rows]


    def get_run_summary(run_id: str) -> Optional[dict[str, Any]]:
        with _conn() as c:
            row = c.execute("SELECT * FROM runs WHERE run_id = ?", (run_id,)).fetchone()
        return dict(row) if row else None

ANNOTATION: Two read helpers used by the UI's "Done" page. `get_events_for_run`
returns events in timestamp order, which is the natural reading order. Both
return plain dicts (not sqlite3.Row objects) so they JSON-serialize cleanly
when offered as an audit-trail download.


--------------------------------------------------------------------------------
FILE: src/progress.py
--------------------------------------------------------------------------------

PURPOSE: A singleton reporter that decouples agents from the UI. Agents emit
events; the UI registers a callback to handle them. Without a callback, emits
are no-ops.

----------------------------------------------------------------
[Block 1: ProgressReporter class]
----------------------------------------------------------------

    class ProgressReporter:
        def __init__(self) -> None:
            self._callback: Optional[Callable[[dict[str, Any]], None]] = None

        def set_callback(self, callback) -> None:
            self._callback = callback

        def clear(self) -> None:
            self._callback = None

        def emit(self, **event: Any) -> None:
            if self._callback is None:
                return
            try:
                self._callback(event)
            except Exception:
                pass

ANNOTATION: A small class with three operations: register, clear, and emit.

The `try / except` around the callback is intentional. Progress reporting is
observability — it tells the UI what's happening — and it must never break
the pipeline. If the UI hits an error rendering progress (which can happen
with Streamlit during reruns), the agent has already done its work; we are
not going to let a UI hiccup take down the run.

If asked "what's the engineering principle behind that try/except?" the
answer is: instrumentation should fail open. Logs and traces should never
be on the critical path of correctness.

----------------------------------------------------------------
[Block 2: module-level singleton and accessor]
----------------------------------------------------------------

    _reporter = ProgressReporter()

    def get_reporter() -> ProgressReporter:
        return _reporter

ANNOTATION: One reporter instance per Python process, accessed via a module-
level function. Agents import `get_reporter` and call `.emit(...)`. The UI
imports `get_reporter` and calls `.set_callback(my_handler)` before kicking
off the pipeline, then `.clear()` afterwards (in a finally block) to prevent
the callback from leaking across Streamlit reruns.

The events agents emit have two kinds today:

  kind='clause_step'  emitted at the top of each iteration of compare, score,
                      and verify. Carries (stage, current, total, clause_type)
                      so the UI can render "clause 14 of 21".

  kind='llm_usage'    emitted by call_llm after every LLM call. Carries
                      (stage, input_tokens, output_tokens). The UI accumulates
                      these into the running cost counter.


--------------------------------------------------------------------------------
FILE: src/llm.py
--------------------------------------------------------------------------------

PURPOSE: Wraps the Anthropic client with audit logging, retries, prompt
versioning, and progress emission. All agents call into this module rather
than the SDK directly. This is the chokepoint that makes the audit story
defensible.

[Block 2: retry decorator and signature]
----------------------------------------------------------------

    @retry(stop=stop_after_attempt(3), wait=wait_exponential(multiplier=1, min=2, max=10))
    def call_llm(
        *,
        run_id: str,
        stage: str,
        system: str,
        user: str,
        max_tokens: int = 2048,
        temperature: float = 0.0,
        expect_json: bool = False,
    ) -> dict[str, Any]:

ANNOTATION: The `@retry` decorator (from tenacity) wraps the function so any
exception from the LLM call triggers up to two retries with exponential
backoff (2s, then up to 10s). After three total attempts, the exception
propagates.

This is the resilience layer for transient API failures: rate limits,
intermittent 5xx responses, network blips. The decorator makes the retry
behavior declarative — no try/except boilerplate inside the function.

The `*,` in the signature forces all arguments to be keyword-only. That makes
call sites self-documenting: you write `call_llm(run_id=..., stage=...)`
rather than relying on positional argument order. Important here because
several arguments are similar strings (run_id and stage are both UUIDs/short
strings) and positional confusion would be hard to spot.

`temperature=0.0` is a deliberate choice for a contract review agent. We want
the output to be as deterministic as possible — same contract, same prompt,
same result. (In practice, even at temperature=0 there is some variance, as
the user observed comparing two runs of the same aggressive contract.)

----------------------------------------------------------------

----------------------------------------------------------------

    response = client.messages.create(
        model=LLM_MODEL,
        max_tokens=max_tokens,
        temperature=temperature,
        system=system,
        messages=[{"role": "user", "content": user}],
    )

    text = "".join(block.text for block in response.content if block.type == "text")
    parsed: Optional[Any] = None
    parse_error: Optional[str] = None

    if expect_json:
        cleaned = text.strip()
        if cleaned.startswith("```"):
            cleaned = cleaned.split("```", 2)[1]
            if cleaned.startswith("json"):
                cleaned = cleaned[4:]
            cleaned = cleaned.rsplit("```", 1)[0].strip()
        try:
            parsed = json.loads(cleaned)
        except json.JSONDecodeError as e:
            parse_error = str(e)

ANNOTATION: One Anthropic API call. The response can contain multiple content
blocks (text, tool use, etc.); we extract just the text blocks and concatenate
them.

The `expect_json` block handles a real-world LLM quirk: even when explicitly
asked for "no prose outside the JSON object", models occasionally wrap their
JSON output in markdown code fences (```json ... ```). The cleanup logic
strips those fences before parsing.

Parse errors are captured into `parse_error` rather than raising. The agent
that called us gets back a `parsed=None, parse_error="..."` result and decides
how to handle it (typically by treating it as a missing-output case via
`.get(key, default)`). Better that than crashing the pipeline on a single
malformed response.

  - The parsed JSON if applicable (for searchability)
  - Token counts (for cost reconstruction)
  - The model identifier and prompt version (for provenance)

The UI's progress callback accumulates input_tokens and output_tokens across
the run and renders a live cost counter. Same data the audit captures, but
surfaced live for the demo.

If asked "how do you reconstruct what the AI did six months ago?" the answer
is: the audit DB has the prompt version that was active, the full prompt that
was sent, and the full response that came back. You can replay the call.

PURPOSE: Embeds the playbook into a local vector store and retrieves the most
relevant playbook positions for any given clause. This is the
retrieval-augmented generation layer the JD specifically asks for.

----------------------------------------------------------------
[Block 1: collection and client helpers]
----------------------------------------------------------------

    _COLLECTION_NAME = "playbook"


    def _client() -> chromadb.PersistentClient:
        return chromadb.PersistentClient(
            path=str(CHROMA_DIR),
            settings=Settings(anonymized_telemetry=False),
        )


    def _embedder():
        return embedding_functions.SentenceTransformerEmbeddingFunction(
            model_name="all-MiniLM-L6-v2"
        )

ANNOTATION: ChromaDB's `PersistentClient` writes to disk so the index survives
restarts. `anonymized_telemetry=False` opts out of Chroma's default telemetry
beacon — a small but meaningful privacy/transparency choice for a project
that handles legal documents.

The embedder uses `all-MiniLM-L6-v2`, a small (~80 MB) sentence-transformer
model that runs on CPU and produces 384-dimensional embeddings. It is not a
legal-domain model, which means the embeddings are general-purpose. For a
production deployment you would replace this with a legal-domain model
(e.g. one fine-tuned on contract corpora) for better retrieval accuracy.

That swap is one line of code, which is part of the design intent. If asked
"how would you improve retrieval?" the answer is "swap the embedder, in the
same way you'd swap an LLM provider."

----------------------------------------------------------------
[Block 2: parse_playbook]
----------------------------------------------------------------

    def _parse_playbook(md: str) -> list[dict[str, str]]:
        """Split a markdown playbook into sections keyed by H2 headings."""
        sections: list[dict[str, str]] = []
        current_title: str | None = None
        current_body: list[str] = []

        for line in md.splitlines():
            if line.startswith("## "):
                if current_title is not None:
                    sections.append(
                        {"title": current_title, "body": "\n".join(current_body).strip()}
                    )
                current_title = line[3:].strip()
                current_body = []
            elif current_title is not None:
                current_body.append(line)

        if current_title is not None:
            sections.append(
                {"title": current_title, "body": "\n".join(current_body).strip()}
            )

        return [s for s in sections if s["body"]]

ANNOTATION: A small markdown parser that splits the playbook into
H2-delimited sections. Each H2 heading becomes a separately-retrievable
section. This is why playbook.md has headings like "## Limitation of
Liability" and "## Indemnification" — they are the chunking boundaries.

Choosing H2 as the chunk boundary is a tradeoff. Smaller chunks (paragraphs)
would give more precise retrieval but lose context; larger chunks (whole
playbook) would lose retrieval entirely. H2 captures one coherent position
per chunk, which is the right granularity for clause comparison.

A real production system would chunk more sophisticatedly — sliding windows
with overlap, or semantic chunking — but for a portfolio demo this is the
simplest thing that works.

----------------------------------------------------------------
[Block 3: index_playbook]
----------------------------------------------------------------

    def index_playbook(playbook_path: Path | None = None) -> int:
        """(Re)build the playbook index. Returns the number of sections indexed."""
        if playbook_path is None:
            playbook_path = PLAYBOOK_DIR / "playbook.md"

        md = playbook_path.read_text(encoding="utf-8")
        sections = _parse_playbook(md)

        client = _client()
        try:
            client.delete_collection(_COLLECTION_NAME)
        except Exception:
            pass

        collection = client.create_collection(
            name=_COLLECTION_NAME,
            embedding_function=_embedder(),
        )

        ids = [
            hashlib.md5(s["title"].encode()).hexdigest() for s in sections
        ]
        documents = [f"{s['title']}\n\n{s['body']}" for s in sections]
        metadatas = [{"title": s["title"]} for s in sections]

        collection.add(ids=ids, documents=documents, metadatas=metadatas)
        return len(sections)

ANNOTATION: This is the function the user calls (via the sidebar button or
CLI) to (re)build the index. It is idempotent — deletes the existing
collection if any, creates a fresh one, embeds every section.

The IDs are MD5 hashes of section titles. That is intentional and useful: if
you re-index after an edit, sections with unchanged titles get stable IDs.
You can also look up a specific section by hashing its title.

The `delete_collection` call is wrapped in a bare except because Chroma
raises a different exception type depending on whether the collection exists.
Swallowing it is acceptable here — we're about to create the collection
either way.

----------------------------------------------------------------
[Block 4: retrieve_playbook_position]
----------------------------------------------------------------

    def retrieve_playbook_position(
        *, run_id: str, clause_text: str, clause_type: str, top_k: int = 2
    ) -> list[dict[str, Any]]:
        client = _client()
        try:
            collection = client.get_collection(
                name=_COLLECTION_NAME,
                embedding_function=_embedder(),
            )
        except Exception:
            index_playbook()
            collection = client.get_collection(
                name=_COLLECTION_NAME,
                embedding_function=_embedder(),
            )

        query = f"Clause type: {clause_type}\n\n{clause_text}"
        results = collection.query(query_texts=[query], n_results=top_k)

        matches = []
        for i in range(len(results["ids"][0])):
            matches.append(
                {
                    "id": results["ids"][0][i],
                    "title": results["metadatas"][0][i]["title"],
                    "document": results["documents"][0][i],
                    "distance": results["distances"][0][i] if results.get("distances") else None,
                }
            )

        log_event(
            run_id=run_id,
            stage="playbook",
            event_type="retrieval",
            payload={
                "query": query,
                "clause_type": clause_type,
                "top_k": top_k,
                "matches": matches,
            },
        )
        return matches

ANNOTATION: This is what the compare agent calls once per clause. The query
includes both the clause type label and the clause text, which biases
retrieval toward sections of the playbook that address the same kind of
clause.

The auto-rebuild fallback (the second try) is a safety net: if you blow away
the audit directory but keep the rest of the project, the next run will
notice the collection is missing and rebuild it before retrieving.

The retrieval is logged to the audit DB with event_type='retrieval'. That's
why the audit log for a typical run shows ~21 retrieval events alongside the
LLM calls — one per clause processed by the compare agent. If asked "did the
right playbook position get retrieved for clause X?" you answer it by
querying the audit log.

PURPOSE: Identifies the contract type. Cheap, fast first stage that lets
downstream agents condition on the type if they want to.

----------------------------------------------------------------
[Block 1: prompt]
----------------------------------------------------------------

    SYSTEM_PROMPT = """You are a legal classifier. Given the text of a contract,
    identify its primary type using a short canonical label.

    Allowed labels (use exactly one):
    - nda
    - msa
    - saas_subscription
    - employment
    - consulting_services
    - licensing
    - distribution
    - procurement
    - other

    Return strict JSON:
    {
      "contract_type": "<label>",
      "confidence": <float 0-1>,
      "reasoning": "<one sentence>"
    }
    No prose outside the JSON object."""

ANNOTATION: Three things to notice about this prompt as a pattern that
recurs across every agent:

1. The label set is ENUMERATED. The model has to pick from a fixed
   vocabulary. This is the simplest way to get reliable downstream behavior:
   you know what values to expect.

2. The output format is STRICT JSON with named fields. Free-form output is
   harder to consume programmatically. The "no prose outside the JSON
   object" line is the most-violated instruction in LLM prompting; we
   defensively handle violation in src/llm.py's parse logic.

3. The prompt does NOT say "if you don't know, take your best guess." That
   is intentional — we want low confidence to produce low confidence scores,
   not false-confident wrong labels.

----------------------------------------------------------------
[Block 2: classify function]
----------------------------------------------------------------

    def classify(state: ReviewState) -> ReviewState:
        contract_text = state["contract_text"]
        user = f"Classify this contract:\n\n{contract_text[:6000]}"

        result = call_llm(
            run_id=state["run_id"],
            stage="classifier",
            system=SYSTEM_PROMPT,
            user=user,
            expect_json=True,
            max_tokens=300,
        )

        parsed = result.get("parsed") or {}
        return {
            **state,
            "contract_type": parsed.get("contract_type", "other"),
            "contract_type_confidence": float(parsed.get("confidence", 0.0)),
        }

ANNOTATION: Several deliberate choices:

- `contract_text[:6000]` truncates the input to the first 6000 characters.
  Contract type is almost always identifiable from the opening (parties,
  recitals, first few sections); paying for the full document here would be
  wasteful.

- `max_tokens=300` is a tight cap. The expected output is a small JSON object
  with three short fields. A tight cap saves money and gives a clean upper
  bound on response time.

- `parsed.get("contract_type", "other")` defaults to "other" if parsing
  failed or the field was missing. The pipeline never crashes on missing
  output; it falls through to a safe default.

- `float(parsed.get("confidence", 0.0))` defensively coerces in case the
  model returned the confidence as a string ("0.95") rather than a number.

The state evolution pattern (`{**state, "new_key": new_value}`) appears in
every agent. Every agent function is a pure(ish) state-to-state mapper.

PURPOSE: Pulls the high-leverage clauses verbatim from the contract. Each

[Block 1: target clauses]

    TARGET_CLAUSES = [
        "limitation_of_liability",

ANNOTATION: The ten clause types the extractor looks for. This list is the

in playbook.md, and every type the extractor finds will be matched against

If you add a new clause type, you do it in two places: this list, and

benefit from validation in production (e.g. a startup check that every

----------------------------------------------------------------
[Block 2: prompt]
----------------------------------------------------------------

    SYSTEM_PROMPT = f"""You are a legal clause extractor. Given a contract,
    find each instance of the following clause types and return their
    verbatim text.

    Clause types to look for:
    {chr(10).join(f"- {c}" for c in TARGET_CLAUSES)}

    Return strict JSON:
    {{
      "clauses": [
        {{
          "clause_type": "<one of the types above>",
          "text": "<verbatim clause text, copied exactly from the contract>",
          "location_hint": "<section number, heading, or other locator>"
        }}
      ]
    }}

    Rules:
    - Quote text VERBATIM. Do not paraphrase or summarise.
    - If a clause type is not present, omit it from the list.
    - If multiple instances exist, include each separately.
    - If no listed clauses are found, return {{"clauses": []}}.
    - No prose outside the JSON object."""

ANNOTATION: The "verbatim" instruction is the most important rule in this
prompt and the source of one of the system's strengths. Because the
extractor is told to quote rather than paraphrase, the verifier can later
use a mechanical string match to detect fabricated quotes (see
src/agents/verifier.py).

If the extractor were allowed to paraphrase, the verifier's mechanical check
wouldn't work, and you'd be back to "ask another LLM to grade the first
LLM's work" with all the unreliability that implies.

The "include each separately" instruction is what produces the granular
extraction we observed in test runs — Section 7.1 and Section 7.2 of the
aggressive contract come out as two separate limitation_of_liability clauses,
not one merged blob. That is correct behavior.

[Block 3: extract function]
----------------------------------------------------------------

    def extract(state: ReviewState) -> ReviewState:
        contract_text = state["contract_text"]
        user = (
            f"Contract type: {state.get('contract_type', 'unknown')}\n\n"
            f"Contract text:\n\n{contract_text}"
        )

        result = call_llm(
            run_id=state["run_id"],
            stage="extractor",
            system=SYSTEM_PROMPT,
            user=user,
            expect_json=True,
            max_tokens=4000,
        )

        parsed = result.get("parsed") or {}
        raw = parsed.get("clauses", [])

        clauses: list[Clause] = []
        for c in raw:
            clauses.append(
                Clause(
                    clause_id=str(uuid.uuid4())[:8],
                    clause_type=c.get("clause_type", "other"),
                    text=c.get("text", "").strip(),
                    location_hint=c.get("location_hint", ""),
                    playbook_matches=[],
                    comparison={},
                    risk_score=0,
                    risk_rationale="",
                )
            )

        return {**state, "clauses": clauses}

ANNOTATION: Notable design choices:

- `max_tokens=4000` is much larger than the classifier's 300. The output
  here is many clauses with verbatim quotes; for a long contract with 20+
  extracted clauses, the response can be substantial.

- Every clause gets a fresh 8-character UUID. Short enough to be readable
  in the UI, long enough to be unique within a run. Used everywhere
  downstream — the HITL form keys on clause_id, the verifier references
  it in findings, and the audit log can correlate events to specific
  clauses through it.

- The `Clause` is initialized with empty fields for what later stages will
  populate (`playbook_matches=[]`, `comparison={}`, etc.). This makes the
  Clause objects "complete enough" that downstream agents can read those
  fields without crashing on missing keys, even if they have no value yet.

- The whole contract is passed to the extractor in `user`, not truncated.
  The classifier gets the first 6000 chars; the extractor needs the whole
  document. This is the most expensive single LLM call in the pipeline
  (one large input).

PURPOSE: For each extracted clause, retrieve the most relevant playbook

----------------------------------------------------------------
[Block 1: prompt]
----------------------------------------------------------------

    SYSTEM_PROMPT = """You compare a contract clause against approved playbook
    positions and identify deviations.

    Return strict JSON:
    {
      "alignment": "aligned" | "partial" | "deviation" | "no_position",
      "deviations": [
        {"issue": "<short description>", "severity": "low" | "medium" | "high"}
      ],
      "suggested_redline": "<redlined alternative text, or empty if aligned>",
      "rationale": "<one or two sentences>"
    }

    Rules:
    - "aligned" only if the clause materially matches the playbook position.
    - "no_position" if the playbook does not address this clause type.
    - Cite playbook language by paraphrase only — do not quote it back verbatim.
    - No prose outside the JSON object."""

ANNOTATION: Several thoughtful constraints in this prompt:

- Four possible alignment values (aligned/partial/deviation/no_position),
  not two. "no_position" lets the agent honestly say "the playbook doesn't
  cover this," which is a useful signal both for users and for the verifier.

- Deviations are tagged with severity. The risk scorer uses these tags as
  one of its inputs.

- "Cite playbook language by paraphrase only" — this is to prevent the
  agent from copying playbook prose into its output verbatim, which would
  bloat the response and confuse the verifier.

- "Suggested redline" is required even when there is no deviation (in which
  case it can be empty). Having a stable schema simplifies downstream code.

[Block 2: compare function — setup]
----------------------------------------------------------------

    def compare(state: ReviewState) -> ReviewState:
        run_id = state["run_id"]
        updated_clauses = []
        clauses = state.get("clauses", [])
        total = len(clauses)

        for i, clause in enumerate(clauses):
            get_reporter().emit(
                kind="clause_step",
                stage="compare",
                current=i + 1,
                total=total,
                clause_type=clause["clause_type"],
            )

ANNOTATION: This is one of the three per-clause loops in the pipeline. The
progress reporter emit at the top of each iteration is what powers the live
"clause 14 of 21 (limitation_of_liability)" sub-update in the UI.

The emit comes BEFORE the work, not after. That way the UI can show
"working on clause 14" while the LLM call is in flight, rather than only
updating after the fact.

----------------------------------------------------------------
[Block 3: per-clause RAG retrieval and LLM call]
----------------------------------------------------------------

            matches = retrieve_playbook_position(
                run_id=run_id,
                clause_text=clause["text"],
                clause_type=clause["clause_type"],
                top_k=2,
            )

            playbook_context = "\n\n---\n\n".join(
                m["document"] for m in matches
            ) or "(no playbook entries retrieved)"

            user = (
                f"Clause type: {clause['clause_type']}\n"
                f"Clause text:\n{clause['text']}\n\n"
                f"Relevant playbook positions:\n{playbook_context}"
            )

            result = call_llm(
                run_id=run_id,
                stage="playbook",
                system=SYSTEM_PROMPT,
                user=user,
                expect_json=True,
                max_tokens=1500,
            )

            parsed = result.get("parsed") or {}
            clause["playbook_matches"] = matches
            clause["comparison"] = {
                "alignment": parsed.get("alignment", "no_position"),
                "deviations": parsed.get("deviations", []),
                "suggested_redline": parsed.get("suggested_redline", ""),
                "rationale": parsed.get("rationale", ""),
            }
            updated_clauses.append(clause)

ANNOTATION: This is the heart of RAG in the pipeline. For each clause:

1. Retrieve the top 2 most relevant playbook sections via vector similarity.
   `top_k=2` is a balance — k=1 risks missing a relevant section if the
   match is borderline; k=5 dilutes the prompt with weakly-related content.

2. Concatenate the retrieved sections (separated by `---`) into a single
   playbook_context string.

3. Build the user prompt: clause type, clause text, and the retrieved
   playbook context.

4. Make the LLM call. Default to "no_position" alignment if parsing fails,
   on the principle that "we don't know" is safer than "we know it's
   aligned."

5. Stash both the retrieved matches AND the comparison output back into
   the Clause object. The matches are kept so the audit trail can show
   exactly which playbook sections influenced the comparison.

The fallback `"(no playbook entries retrieved)"` string handles the edge
case where retrieval returned nothing — rare but possible if the playbook
hasn't been indexed yet. Without the fallback, the `or` would leave the
prompt with an empty section that confuses the model.

----------------------------------------------------------------
[Block 4: return]
----------------------------------------------------------------

        return {**state, "clauses": updated_clauses}

ANNOTATION: Standard state evolution. The new state has the same Clause
objects, but each is now enriched with playbook_matches and comparison
fields. Downstream agents (score, verify) read these fields.

FILE: src/agents/risk_scorer.py
--------------------------------------------------------------------------------

PURPOSE: Assigns a 0-100 risk score to each clause, then computes an overall
contract risk score that explicitly considers clause interactions rather
than averaging.

----------------------------------------------------------------
[Block 1: per-clause prompt]
----------------------------------------------------------------

    PER_CLAUSE_SYSTEM = """You assign a risk score (0-100) to a contract clause.

    Inputs you receive: the clause text and a playbook-comparison summary.

    Scoring guide:
    - 0-20: aligned with playbook, low risk
    - 21-50: minor deviations, manageable
    - 51-75: material deviation, negotiation recommended
    - 76-100: severe deviation or unusual exposure, escalate

    Return strict JSON:
    {
      "risk_score": <int 0-100>,
      "rationale": "<one or two sentences>"
    }
    No prose outside the JSON object."""

ANNOTATION: The 0-100 scale was chosen over a categorical (low/medium/high)
or a small-integer (1-5) alternative for two reasons:

1. The HITL threshold is a single number compared against a single number.
   `risk >= 60` is simpler than mapping categories to thresholds.

2. Per-clause scores need to combine in a meaningful way for the overall
   score. A 0-100 scale gives the overall scorer continuous values to
   reason about.

The buckets in the scoring guide (0-20, 21-50, etc.) anchor the model's
output to a stable distribution rather than letting it drift freely.

Known weakness: the per-clause scorer tends to undershoot on extreme
deviations. A clause that caps vendor liability at $1,000 should arguably
score 95+, but the model has consistently scored it in the high 80s.
This is a calibration issue worth surfacing in interview discussion —

ANNOTATION: The "NOT a simple average" instruction is doing real work. A

This is the prompt that produces the sophisticated overall rationales we
observed in test runs ("severe deviations compound across liability,
indemnification, IP, data protection, and dispute resolution to create
catastrophic financial, legal, and operational exposure"). The model is
good at this kind of structured reasoning when explicitly told to do it.

better to under-score and miss escalating than to over-score and trigger
false HITL pauses. A real production system might log a verifier finding

PURPOSE: Independently checks the agents' outputs against the source contract
to detect hallucination. This is the single most important agent in the
pipeline for the JD's "hallucination detection and mitigation" requirement.

----------------------------------------------------------------
[Block 1: prompt]
----------------------------------------------------------------

    SYSTEM_PROMPT = """You are an independent verifier checking an AI's contract
    review for accuracy and hallucination.

    You are given a clause as quoted by the AI and the corresponding source text
    from the contract. Determine whether the quote is faithful (allowing for
    trivial whitespace differences only) and whether the AI's stated rationale is
    supported.

    Return strict JSON:
    {
      "quote_faithful": true | false,
      "rationale_supported": true | false,
      "issue": "<short description, or empty string if no issue>",
      "severity": "low" | "medium" | "high"
    }
    No prose outside the JSON object."""

ANNOTATION: The verifier checks two distinct things:

1. quote_faithful — does the AI's "verbatim" quote actually match the
   source contract? This catches the most common hallucination, where the
   model paraphrases or invents content while claiming to quote.

2. rationale_supported — does the AI's stated rationale follow from the
   clause text and the playbook? This catches the more subtle failure
   where the quote is real but the analysis doesn't match.

Crucially, the verifier is independent — it does not see the playbook
comparison output's verdict before judging. It receives the AI's rationale
as a fact to check, not a conclusion to defer to. That independence is
what makes its disagreement meaningful.

----------------------------------------------------------------
[Block 2: mechanical fidelity helpers]
----------------------------------------------------------------

    def _normalize(text: str) -> str:
        return re.sub(r"\s+", " ", text).strip().lower()


    def _quote_in_source(quote: str, source: str) -> bool:
        """Coarse fidelity check — exact match after whitespace normalization."""
        return _normalize(quote) in _normalize(source)

ANNOTATION: This is the "no-LLM" first-pass check that catches the most
common hallucination cheaply. Normalize both the quote and the source
(collapse whitespace, lowercase), then check substring containment.

If the quote isn't in the source verbatim, the AI either fabricated it
or paraphrased something it claimed to quote — either way, a high-severity
finding without spending an LLM call.

A real production version would handle minor punctuation differences,
quote-mark variations, and OCR errors more gracefully. For a portfolio
demo, the strict-match-after-normalization is the right balance of
simplicity and effectiveness.

----------------------------------------------------------------
[Block 3: verify function — per-clause loop with mechanical check]
----------------------------------------------------------------

    def verify(state: ReviewState) -> ReviewState:
        run_id = state["run_id"]
        contract_text = state["contract_text"]
        findings: list[VerifierFinding] = []
        clauses = state.get("clauses", [])
        total = len(clauses)

        for i, clause in enumerate(clauses):
            get_reporter().emit(
                kind="clause_step",
                stage="verify",
                current=i + 1,
                total=total,
                clause_type=clause["clause_type"],
            )

            # Cheap mechanical check first
            if not _quote_in_source(clause["text"], contract_text):
                findings.append(
                    VerifierFinding(
                        clause_id=clause["clause_id"],
                        issue="Extracted quote not found verbatim in source contract",
                        severity="high",
                        quote_from_ai=clause["text"][:200],
                        quote_from_source=None,
                    )
                )
                continue

ANNOTATION: The mechanical check runs first, before any LLM call, and
short-circuits with `continue` if it triggers. There is no point asking
an LLM to verify a quote that demonstrably doesn't exist in the source —
we already know the answer.

When the mechanical check fires, severity is hardcoded to "high" because
fabricated quotes are categorically the worst kind of hallucination. They
cannot be excused as a misunderstanding; they are made-up content
presented as fact.

Note `quote_from_ai=clause["text"][:200]` truncates to 200 characters for
the finding payload. The full quote is already in the audit log via the
extractor's LLM call; we don't need to duplicate it in full here.

----------------------------------------------------------------
[Block 4: LLM rationale check]
----------------------------------------------------------------

            # LLM-based rationale check
            comparison = clause.get("comparison", {})
            user = (
                f"AI-extracted clause text:\n{clause['text']}\n\n"
                f"AI's comparison rationale: {comparison.get('rationale', '')}\n"
                f"AI's alignment verdict: {comparison.get('alignment', '')}\n\n"
                f"Source contract excerpt (verbatim):\n"
                f"{_find_source_excerpt(clause['text'], contract_text)}"
            )

            result = call_llm(
                run_id=run_id,
                stage="verifier",
                system=SYSTEM_PROMPT,
                user=user,
                expect_json=True,
                max_tokens=500,
            )
            parsed = result.get("parsed") or {}

            if not parsed.get("quote_faithful", True) or not parsed.get(
                "rationale_supported", True
            ):
                findings.append(
                    VerifierFinding(
                        clause_id=clause["clause_id"],
                        issue=parsed.get("issue", "Unspecified verifier finding"),
                        severity=parsed.get("severity", "medium"),
                        quote_from_ai=clause["text"][:200],
                        quote_from_source=None,
                    )
                )

ANNOTATION: For every clause that survived the mechanical check, the LLM
verifier gets:

- The clause text the AI extracted
- The rationale and alignment verdict the comparison agent produced
- A relevant excerpt of the source contract

It returns a structured judgment. A finding is recorded only if the verifier
flagged either the quote or the rationale as unsupported — note the default
values in `.get("quote_faithful", True)`. If parsing failed entirely, both
checks default to True (no finding), on the principle that a parse failure
is not in itself evidence of hallucination.

The defaults bias the verifier toward false negatives (missing real issues)
rather than false positives (flagging clean output). For a HITL system, that
is the right bias: false negatives are caught by the threshold-based gate
or by the human reviewing every flagged clause; false positives erode trust
in the system.

----------------------------------------------------------------
[Block 5: HITL gate trigger]
----------------------------------------------------------------

        # HITL gate trigger
        has_high_severity = any(f["severity"] == "high" for f in findings)
        risk_above_threshold = (
            state.get("overall_risk_score", 0) >= _hitl_threshold()
        )
        requires_human_review = has_high_severity or risk_above_threshold

        return {
            **state,
            "verifier_findings": findings,
            "requires_human_review": requires_human_review,
        }

ANNOTATION: This is where the HITL gate decision is computed. Two
independent triggers:

  has_high_severity     — any verifier finding tagged as "high" forces
                          human review, regardless of overall risk score.
                          A fabricated quote should not slip through just
                          because the rest of the contract scored well.

  risk_above_threshold  — the overall risk score is at or above the
                          configured threshold (default 60). This catches
                          high-risk contracts that the verifier didn't
                          flag.

The two triggers are independent and combined with OR. Either is sufficient
to force human review; both becoming false is necessary to skip the gate.

If asked "what would it take to remove the human from the loop?" the
answer is: nothing in this pipeline. The threshold and the verifier are
two safety mechanisms operating on different failure modes. The right
design isn't to remove the human; it's to lower the threshold over time
as confidence grows and instrument every override so the system learns.

----------------------------------------------------------------
[Block 6: helpers]
----------------------------------------------------------------

    def _hitl_threshold() -> int:
        from config import HITL_RISK_THRESHOLD
        return HITL_RISK_THRESHOLD


    def _find_source_excerpt(quote: str, source: str, window: int = 300) -> str:
        """Return a window of source text around the (normalized) quote location."""
        norm_source = _normalize(source)
        norm_quote = _normalize(quote)
        idx = norm_source.find(norm_quote)
        if idx == -1:
            return source[:window * 2]
        approx_start = max(0, int(idx * len(source) / max(len(norm_source), 1)) - window)
        approx_end = min(len(source), approx_start + len(quote) + window * 2)
        return source[approx_start:approx_end]

ANNOTATION: `_hitl_threshold` is a function (not just an import at the top
of the file) so the threshold is read every time, which means the agent
picks up environment variable changes between runs. Small detail but it
matters in long-running processes where you might tune the threshold
without restarting.

`_find_source_excerpt` extracts a windowed excerpt of the source text
around where the quote appears. The excerpt is what gets sent to the
LLM verifier as "source contract excerpt (verbatim)". Sending the whole
contract every time would be wasteful; sending nothing would deprive the
verifier of context.

The mapping from normalized indices back to original indices (the
`int(idx * len(source) / max(len(norm_source), 1))` line) is approximate
because normalization can change string length. It's good enough for
finding a window — we don't need character-perfect alignment.

FILE: src/orchestrator.py
--------------------------------------------------------------------------------

PURPOSE: Wires the agents into a LangGraph state graph, defines the routing
logic between them, configures the HITL interrupt point, and produces the
final report. This is the file that turns five independent agents into a
pipeline.

----------------------------------------------------------------
[Block 1: routing decision after verify]
----------------------------------------------------------------

    def _route_after_verify(state: ReviewState) -> Literal["hitl", "finalize"]:
        return "hitl" if state.get("requires_human_review") else "finalize"

ANNOTATION: This is a LangGraph "conditional edge" function. After the
verifier runs, this function is called with the current state and returns
a string naming the next node. Two possibilities:

  - "hitl" if requires_human_review is True (set by the verifier based on
    its findings or the risk threshold)
  - "finalize" otherwise

The routing function itself does not act on the state — it just returns
the name of the next node. LangGraph's edge configuration translates that
name into a transition.

----------------------------------------------------------------
[Block 2: _hitl_node]
----------------------------------------------------------------

    def _hitl_node(state: ReviewState) -> ReviewState:
        decision = state.get("hitl_decision")
        if decision:
            log_event(
                run_id=state["run_id"],
                stage="hitl",
                event_type="human_decision",
                payload=dict(decision),
            )
            modified = decision.get("modified_clauses", {}) or {}
            if modified:
                updated = []
                for c in state.get("clauses", []):
                    override = modified.get(c["clause_id"])
                    if override:
                        c = {**c, **override}
                    updated.append(c)
                state = {**state, "clauses": updated}
        return state

ANNOTATION: The HITL node has a subtle structure. It does almost nothing on
the way in (because the graph is configured to PAUSE before this node
executes via interrupt_before — see build_graph below). The state at that
pause point has no `hitl_decision` field yet.

What happens then is:
  1. The graph pauses with state persisted to SQLite.
  2. The UI reads the paused state.
  3. The user submits a decision in the UI.
  4. The UI calls `graph.update_state()` to write the decision into state.
  5. The UI calls `graph.stream(None, ...)` to resume.
  6. THIS NODE NOW RUNS, with hitl_decision present in state.

When the node finally executes, it does two things:

  a. Logs a 'human_decision' event to the audit DB. This is the entry that
     records who approved/rejected/modified the contract, what notes they
     left, and what overrides they made. It is the most important entry in
     the audit log for any post-hoc review.

  b. Applies clause-level overrides. For each `clause_id` in
     `modified_clauses`, finds the matching clause in state and merges the
     override fields into it. Most commonly, this is bumping a risk_score.

The merge `c = {**c, **override}` makes the override fields take
precedence over the AI's values. The resulting clause object reflects the
human's correction, and that is what goes into the final report.

----------------------------------------------------------------
[Block 3: _finalize — header]
----------------------------------------------------------------

    def _finalize(state: ReviewState) -> ReviewState:
        lines = []
        lines.append(f"# Contract Review Report")
        lines.append("")
        lines.append(f"**Contract:** {state.get('contract_name')}")
        lines.append(f"**Type:** {state.get('contract_type')} "
                     f"(confidence {state.get('contract_type_confidence', 0):.2f})")
        lines.append(f"**Overall risk score:** {state.get('overall_risk_score')}/100")
        lines.append(f"**Overall rationale:** {state.get('overall_risk_rationale')}")
        decision = state.get("hitl_decision")
        if decision:
            lines.append(f"**Human review:** {decision.get('decision')} "
                         f"by {decision.get('reviewer')}")
            if decision.get("notes"):
                lines.append(f"  Notes: {decision['notes']}")
        lines.append("")

ANNOTATION: The finalize node assembles the human-readable Markdown report.
The header section captures: contract name, type and confidence, overall
risk score and rationale, and (if there was human review) the reviewer's
decision and notes.

The pattern of building up `lines` and joining at the end is a very
common idiom for generating multi-line text. Cleaner than concatenating
strings with `\n`.

The report is purely deterministic — no LLM call at this stage. By this
point all the AI work is done; finalize just formats what's already in
state.

            lines.append(f"_Location: {c.get('location_hint', 'n/a')}_")

----------------------------------------------------------------

    def build_graph():
        g = StateGraph(ReviewState)

        g.add_node("classify", classify)
        g.add_node("extract", extract)
        g.add_node("compare", compare)
        g.add_node("score", score)
        g.add_node("verify", verify)
        g.add_node("hitl", _hitl_node)
        g.add_node("finalize", _finalize)

        g.set_entry_point("classify")
        g.add_edge("classify", "extract")
        g.add_edge("extract", "compare")
        g.add_edge("compare", "score")
        g.add_edge("score", "verify")
        g.add_conditional_edges("verify", _route_after_verify, {
            "hitl": "hitl",
            "finalize": "finalize",
        })
        g.add_edge("hitl", "finalize")
        g.add_edge("finalize", END)

        conn = sqlite3.connect(str(CHECKPOINT_DB_PATH), check_same_thread=False)
        checkpointer = SqliteSaver(conn)

        return g.compile(
            checkpointer=checkpointer,
            interrupt_before=["hitl"],
        )

ANNOTATION: This is the function that defines the entire pipeline shape.
Reading it top to bottom:

1. Create a StateGraph parameterized on the ReviewState type.

2. Register every agent as a node. The string name (e.g. "classify") is
   what edges reference; the function (e.g. `classify`) is what runs.

3. Set the entry point — execution begins at the classifier.

4. Add unconditional edges between sequential stages: classify→extract,
   extract→compare, compare→score, score→verify.

5. Add the conditional edge after verify. The dict tells LangGraph that
   when `_route_after_verify` returns "hitl", go to the hitl node; when
   it returns "finalize", go to the finalize node.

6. After hitl, the unconditional edge to finalize. There is no path that
   bypasses finalize; every run that begins must produce a final report.

7. The END sentinel marks the end of the graph.

8. Configure the SQLite checkpointer. `check_same_thread=False` allows
   the connection to be used from multiple threads, which Streamlit
   requires.

9. Compile the graph with `interrupt_before=["hitl"]`. This is the magic
   that makes HITL pause/resume work. The graph will execute up to (but
   not including) any node named in `interrupt_before`, persist its state
   to the checkpointer, and yield control. To resume, call
   `graph.stream(None, config)` with the same thread_id; LangGraph will
   pick up where it left off.

If asked "how does pause-and-resume actually work?" the answer is:
LangGraph keeps the entire state machine state in SQLite, indexed by
thread_id. Resuming is just rehydrating that state and continuing from
the next node.

[Block 7: phase 1 — the live progress panel]
----------------------------------------------------------------

The status panel block is the largest single change in the recent update.
The full code is approximately 90 lines; the structure is:

    STAGES = ["classifier", "extractor", "compare", "score", "verify"]
    STAGE_LABELS = { ... human-readable label per stage ... }

    with st.status("Running review pipeline...", expanded=True) as status_panel:
        stage_slots = {s: st.empty() for s in STAGES}
        cost_slot = st.empty()

        # Initialize all stages as pending
        for s in STAGES:
            stage_slots[s].markdown(f"⏳ &nbsp; {STAGE_LABELS[s]}")

        totals = {"in": 0, "out": 0, "calls": 0}
        active = {"stage": None}
        details: dict[str, str] = {}

        def render_cost():
            cost = (totals["in"] / 1_000_000 * PRICE_PER_M_INPUT
                    + totals["out"] / 1_000_000 * PRICE_PER_M_OUTPUT)
            cost_slot.markdown(
                f"💰 &nbsp; **{totals['calls']} LLM calls** · "
                f"{totals['in']:,} in / {totals['out']:,} out tokens · "
                f"~${cost:.3f} so far"
            )

        def render_stage(stage: str, status_icon: str = "🔄"):
            detail = details.get(stage, "")
            suffix = f" — {detail}" if detail else ""
            stage_slots[stage].markdown(
                f"{status_icon} &nbsp; {STAGE_LABELS[stage]}{suffix}"
            )

        def progress_callback(event: dict):
            kind = event.get("kind")
            if kind == "llm_usage":
                totals["in"] += event.get("input_tokens", 0)
                totals["out"] += event.get("output_tokens", 0)
                totals["calls"] += 1
                render_cost()
            elif kind == "clause_step":
                stage = event["stage"]
                details[stage] = (
                    f"clause {event['current']} of {event['total']} "
                    f"({event['clause_type']})"
                )
                render_stage(stage, status_icon="🔄")

        get_reporter().set_callback(progress_callback)
        render_cost()

ANNOTATION: This is the heart of the live UI. Several pieces interact:

- `st.status(...)` creates a collapsible "running" panel. While it is
  open, anything rendered inside it appears as a child element. When the
  pipeline finishes, status_panel.update() changes the label and visual
  state.

- `st.empty()` creates a placeholder element whose contents can be
  replaced in place. The five stage_slots and the cost_slot are these
  placeholders. Initial content is set, then progress_callback updates
  them in place as the pipeline runs.

- `totals` and `details` are mutable dicts captured by closure. The
  callback reads and writes them to track running state.

- `progress_callback` is the function registered with the progress
  reporter. It receives every emit() the agents fire and translates them
  into UI updates. Two event kinds: `llm_usage` (every LLM call) and
  `clause_step` (every per-clause iteration in compare/score/verify).

- `render_cost()` recomputes the total cost from accumulated tokens and
  rewrites the cost_slot. Called after every llm_usage event, so the cost
  ticks up live.

- `render_stage()` updates a stage's line with an icon (⏳ pending,
  🔄 running, ✅ done) and an optional detail (e.g. "clause 14 of 21").
  Called both from the per-clause callback and from the post-node update
  loop below.

[Block 8: phase 1 — driving the graph and updating per-stage status]
----------------------------------------------------------------

        try:
            for update in graph.stream(initial, config=config, stream_mode="updates"):
                for node_name, node_state in update.items():
                    if node_name in STAGES:
                        if active["stage"] and active["stage"] != node_name:
                            render_stage(active["stage"], status_icon="✅")
                        active["stage"] = node_name
                        if node_name == "classifier":
                            ct = node_state.get("contract_type", "?")
                            conf = node_state.get("contract_type_confidence", 0)
                            details[node_name] = f"{ct} (confidence {conf:.2f})"
                        elif node_name == "extractor":
                            n = len(node_state.get("clauses", []) or [])
                            details[node_name] = f"{n} clauses found"
                        render_stage(node_name, status_icon="✅")
                        current = st.session_state.state or {}
                        st.session_state.state = {**current, **node_state}

            for s in STAGES:
                render_stage(s, status_icon="✅")

            snapshot = graph.get_state(config)
            if snapshot.values:
                st.session_state.state = snapshot.values

            if snapshot.next and "hitl" in snapshot.next:
                status_panel.update(label="Pipeline paused — human review required",
                                    state="complete")
            else:
                status_panel.update(label="Pipeline complete", state="complete")
        except Exception as e:
            status_panel.update(label=f"Pipeline failed: {e}", state="error")
            raise
        finally:
            get_reporter().clear()

ANNOTATION: The streaming loop. `stream_mode="updates"` yields one entry
per node completion, which gives a clean "this stage just finished"
signal. Inside the loop:

- For each completed node, mark the previous stage as ✅ and the current
  one as ✅ too (since by the time we see this update, the node has
  finished). Add a brief result detail where useful (contract type,
  clauses found).

- Accumulate the streamed updates into session state for downstream UI
  consumption.

After the stream ends, fetch the authoritative snapshot from the
checkpointer (because stream_mode='updates' yields partial updates, not
the full merged state). Then check whether the graph paused at the HITL
gate (`snapshot.next` will contain "hitl") and update the status panel
label accordingly.

The `try / except / finally` is critical:
  - try: catch any exception during the run
  - except: visually mark the panel as failed (state="error" turns it red)
            and re-raise so Streamlit can show the traceback
  - finally: clear the progress callback unconditionally, so it doesn't
             leak into the next rerun and try to write to a stale UI slot

  - 88 events total

--------------------------------------------------------------------------------
FILE: data/playbook/playbook.md
--------------------------------------------------------------------------------

PURPOSE: The single source of truth for what the playbook RAG retrieves.
Each H2 heading in this file becomes one searchable playbook section.

STRUCTURE: Ten H2 sections, one per clause type that the extractor looks
for. Each section follows the same internal pattern:

    ## Limitation of Liability

    **Approved position:** [normative description of the standard
    company position on this clause]

    **Walkaway terms:**
    - [conditions that would cause the company to walk away from a deal]

    **Talking points:** [optional negotiation context]

ANNOTATION: A few things about how this file is designed:

The "Approved position" / "Walkaway terms" / "Talking points" structure is
deliberately repetitive across sections. Repetition makes the embeddings
more semantically distinct (each section's "approved position" content is
different but has consistent context around it), which helps retrieval.

The walkaway terms are explicit so the comparison agent can identify
walkaway-level violations. Phrases like "uncapped liability for the
customer" and "cap below 6 months of fees" appear verbatim in the
deviations the agent flags during runs.

For a real engagement, this file would be replaced with a client-specific
playbook. The pattern (markdown, H2 sections, position/walkaway/talking
structure) is portable; the content is not. Updating the playbook is
the most important per-client operation in any deployment of this
system.

To update: edit playbook.md, then click "(Re)build playbook index" in the
sidebar. The new content is embedded and replaces the old index. No
restart needed.

================================================================================
SECTION 7: KNOWN LIMITATIONS AND LIKELY INTERVIEW PROBES
================================================================================

Things to know about the system that an experienced interviewer might
press on. Honest acknowledgments are stronger than polished evasions —
the goal is to demonstrate that you know the limits of what you built,
not to claim it has no limits.

LIMITATION 1: PER-CLAUSE SCORE CALIBRATION
The risk scorer tends to undershoot on extreme single-sided deviations.
A clause capping vendor liability at $1,000 should arguably score 95+
but scores in the high 80s. This is a prompt-engineering issue
(the scoring prompt weights number-of-deviations more than severity-of-
each-deviation). Fix: revise the per-clause scoring prompt to weight
severity-when-extreme more heavily, and re-test. Mention this unprompted
in interviews — it shows you read your own outputs critically.

LIMITATION 2: VARIANCE BETWEEN RUNS
Even with temperature=0, two runs of the same contract produce slightly
different outputs. Clause scores can shift ±10 points; verifier findings
can appear in one run and not another. This is a fact about LLM-based
systems, not a bug in this specific pipeline. The architecture mitigates
by using overall risk scores (which are stable) for HITL decisions, not
individual clause scores (which fluctuate).

LIMITATION 3: VERIFIER FALSE NEGATIVE RATE
The mechanical fidelity check is solid (it correctly flags fabricated
quotes). The LLM rationale check is softer — it tends to defer to the
upstream agent's judgment when the rationale is at all plausible. A
production system would want a more confrontational verifier prompt or
a separate evaluation harness measuring verifier recall on synthetic
hallucinations.

LIMITATION 4: PLAYBOOK COVERAGE
The system is bounded by the playbook. Clause types not in TARGET_CLAUSES
(e.g. unusual provisions like "vendor may amend by website posting" in
Section 12.3 of the aggressive contract) are silently ignored. In a real
deployment, playbook curation is itself an operational discipline — and
adding a "general clauses we should review" pass that catches anything
flagged as unusual would be a meaningful improvement.

LIMITATION 5: NO PDF/DOCX INGESTION
The pipeline accepts plain text only. Real contracts arrive as PDFs and
DOCXs. Extending to those formats requires a separate ingestion layer
(text extraction, OCR for scanned documents, table handling) that is
deliberately out of scope here. If asked, the answer is: ingestion is a
solved problem (libraries like pdfplumber, python-docx, AWS Textract);
not solving it here was a scope decision, not a capability one.

LIMITATION 6: NO MULTI-TENANT ISOLATION
Single-user demo. No SSO, no role-based access, no tenant boundaries on
the audit DB. Production would need all of those, plus a proper secrets
manager (AWS Secrets Manager / Azure Key Vault), a deployment pipeline,
and observability (logs, metrics, traces). The architecture supports
those additions without needing to be re-thought; they were just not
built.

LIMITATION 7: 5-MINUTE PIPELINE RUNTIME
The aggressive contract takes ~5 minutes end-to-end because it produces
~21 clauses and each goes through 3 sequential LLM calls (compare, score,
verify). Optimizations not yet applied: parallelizing per-clause work
(LangGraph supports `Send` for fan-out), batching multiple clauses into
single LLM calls, caching playbook retrievals across runs. Worth
mentioning if pressed on performance.


LIKELY INTERVIEW PROBE 1: "How do you guarantee the audit log is complete?"
Answer: Every LLM call goes through src/llm.py:call_llm, which logs to
the audit DB before returning. There is no path that calls Anthropic
directly. If a developer added one in a code review, that's a code review
failure — but architecturally, the chokepoint is enforced.

LIKELY INTERVIEW PROBE 2: "What happens when the AI is wrong?"
Answer: Three layers of defense. (1) The verifier independently checks
quotes and rationales against the source contract. (2) The HITL gate
forces human review when risk crosses a threshold OR when verifier finds
high-severity issues. (3) The audit log records everything, so post-hoc
review can identify systematic errors. Walk through the first-run audit
JSON if available — that's the example where the verifier caught a real
hallucination.

LIKELY INTERVIEW PROBE 3: "Why LangGraph instead of CrewAI/AutoGen?"
Answer: This pipeline is a state machine with conditional routing,
deterministic node ordering, and a hard requirement to pause and resume
across processes. LangGraph's checkpointer plus interrupt_before model
that exactly. CrewAI is optimized for agents that figure out their own
order; AutoGen for multi-turn conversations. Both would mean fighting
the framework here.

LIKELY INTERVIEW PROBE 4: "How would you scale this to 10,000 contracts a month?"
Answer: Three things. (1) Parallelize per-clause work (LangGraph Send for
fan-out within compare/score/verify). (2) Move from SQLite to Postgres
for both audit and checkpointing. (3) Move the LLM provider to enterprise
hosting (Azure OpenAI / AWS Bedrock) for higher rate limits. The cost
math: ~$0.33 per contract × 10,000 = $3,300/month in LLM spend, well
inside any reasonable per-client budget. The bottleneck would be human
review capacity, not AI throughput.

LIKELY INTERVIEW PROBE 5: "What about prompt injection from the contract?"
Answer: A real concern for any system that processes adversarial input.
Current state: the contract text is passed as user content, the agent
instructions are in system prompts. That is the standard mitigation
pattern but is not bulletproof. Production hardening would include:
(a) explicit instructions in the system prompt to ignore embedded
instructions, (b) input sanitization to detect prompt-injection patterns,
(c) running the verifier with a stronger "is this output consistent with
its inputs?" check. Worth being explicit that this is a known threat
model with partial defense.

LIKELY INTERVIEW PROBE 6: "How do you handle a new client with a different playbook?"
Answer: Playbook is content, not code. Replace data/playbook/playbook.md
with the client's playbook, click "Rebuild index," and the system runs
against their positions. For very different playbooks, you might also
adjust TARGET_CLAUSES in the extractor to look for client-specific
clause types. The architecture supports per-client playbooks; the only
extension is making "which playbook" a runtime parameter rather than a
file-path constant.

LIKELY INTERVIEW PROBE 7: "Why did you build this?"
Answer: The legal industry is moving from AI pilots to operational
infrastructure, and the gap between the two is governance. An AI that
drafts a redline is useful; an AI that drafts a redline you can defend
in a regulatory audit is a different category of system. This project
is a working sketch of what the second category looks like — small,
honest, every control real and tested rather than promised.
